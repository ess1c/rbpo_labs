# Как работает сервис - Объяснение для преподавателя

## Архитектура сервиса

### 1. Аутентификация и авторизация

**Механизм JWT токенов:**

1. **Регистрация** (`POST /api/auth/register`)
   - Пользователь создает аккаунт с валидным паролем
   - Пароль хешируется через BCrypt
   - Пользователь сохраняется в БД с ролью USER

2. **Вход** (`POST /api/auth/login`)
   - Пользователь отправляет username и password
   - Система проверяет учетные данные
   - Генерируется пара токенов:
     - **Access Token** (15 минут) - для доступа к API
     - **Refresh Token** (7 дней) - для обновления access токена
   - Создается сессия в БД (`user_sessions`) со статусом ACTIVE

3. **Использование Access Token**
   - Клиент отправляет запросы с заголовком: `Authorization: Bearer {accessToken}`
   - `JwtAuthenticationFilter` извлекает токен, валидирует его и устанавливает аутентификацию
   - Spring Security проверяет права доступа по ролям

4. **Обновление токенов** (`POST /api/auth/refresh`)
   - Клиент отправляет старый refresh токен
   - Система проверяет, что сессия активна и не истекла
   - Старая сессия помечается как REVOKED
   - Создается новая сессия с новым refresh токеном
   - Возвращается новая пара токенов

5. **Безопасность refresh токенов**
   - Старый refresh токен нельзя использовать повторно
   - При попытке использовать отозванный токен возвращается 401/403
   - Сессии автоматически помечаются как EXPIRED при истечении срока

### 2. Управление сессиями

**Модель данных:**
- `UserSession` - хранит информацию о сессии пользователя
- `SessionStatus` - enum со статусами: ACTIVE, REVOKED, EXPIRED
- Каждая сессия связана с пользователем и содержит refresh токен

**Жизненный цикл сессии:**
1. Создание: при логине создается сессия со статусом ACTIVE
2. Обновление: при refresh старая сессия → REVOKED, новая → ACTIVE
3. Истечение: автоматически помечается как EXPIRED при истечении срока

### 3. Структура JWT токенов

**Access Token содержит:**
```json
{
  "userId": 1,
  "username": "testuser",
  "email": "test@example.com",
  "role": "USER",
  "type": "access",
  "sub": "testuser",
  "iat": 1234567890,
  "exp": 1234568790
}
```

**Refresh Token содержит:**
```json
{
  "userId": 1,
  "username": "testuser",
  "sessionId": "uuid-сессии",
  "type": "refresh",
  "sub": "testuser",
  "iat": 1234567890,
  "exp": 1234574490
}
```

### 4. HTTPS и сертификаты

**Цепочка сертификатов:**
```
Root CA (самоподписанный)
  └── Intermediate CA (подписан Root CA)
      └── Server Certificate (подписан Intermediate CA)
```

**Особенности:**
- Все сертификаты содержат идентификатор студента в поле Subject Alternative Name
- Имена сертификатов уникальны (не совпадают с примерами)
- Keystore хранится в `src/main/resources/keystore.p12`
- Пароли хранятся в переменных окружения или секретах CI/CD

**Настройка HTTPS:**
- Spring Boot автоматически включает HTTPS при наличии keystore
- Порт остается 8080, но протокол меняется на HTTPS
- Браузер требует установки корневого CA в доверенные

### 5. CI/CD Pipeline

**GitHub Actions:**
1. **Build** - компиляция проекта
2. **Test** - запуск тестов
3. **Package** - создание JAR файла
   - Если есть секреты keystore, создается версия с HTTPS
   - Артефакты загружаются в GitHub

**Секреты:**
- `KEYSTORE_BASE64` - keystore в base64
- `KEYSTORE_PASSWORD` - пароль keystore
- `DB_USERNAME`, `DB_PASSWORD` - данные БД

**Безопасность:**
- Приватные ключи никогда не попадают в Git
- Keystore хранится только в секретах
- В репозитории только публичные сертификаты

## Потоки данных

### Поток аутентификации:
```
Клиент → POST /auth/login → Проверка credentials → Генерация токенов → 
Создание сессии в БД → Возврат токенов клиенту
```

### Поток доступа к API:
```
Клиент → Запрос с Access Token → JwtAuthenticationFilter → 
Валидация токена → Извлечение пользователя → Установка SecurityContext → 
Проверка прав → Выполнение запроса
```

### Поток обновления токенов:
```
Клиент → POST /auth/refresh с Refresh Token → Проверка сессии → 
Отзыв старой сессии → Создание новой сессии → Генерация новых токенов → 
Возврат новой пары токенов
```

## Защита от атак

1. **Replay атаки**: Refresh токены одноразовые, нельзя использовать повторно
2. **Token theft**: Access токены имеют короткий срок жизни (15 минут)
3. **Session hijacking**: Сессии отслеживаются в БД, можно отозвать
4. **Man-in-the-middle**: HTTPS защищает трафик
5. **CSRF**: Отключен для API (используется stateless JWT)

## Проверка работы

### Тест 1: Полный цикл аутентификации
1. Регистрация → Создан пользователь
2. Логин → Получены токены, создана сессия ACTIVE
3. Запрос с Access Token → Успешный доступ
4. Refresh → Новые токены, старая сессия REVOKED
5. Повторный refresh со старым токеном → Ошибка 401

### Тест 2: Проверка сессий в БД
```sql
SELECT id, user_id, status, created_at, expires_at, revoked_at 
FROM user_sessions;
```
Должны быть видны все сессии с корректными статусами.

### Тест 3: Проверка HTTPS
```bash
curl -k https://localhost:8080/api/categories
```
С флагом `-k` игнорируется проверка сертификата. Без флага должна быть ошибка, если сертификат не установлен в доверенные.

## Ключевые файлы

- `JwtTokenProvider.java` - генерация и валидация токенов
- `TokenService.java` - бизнес-логика работы с токенами
- `JwtAuthenticationFilter.java` - фильтр для проверки токенов
- `UserSession.java` - модель сессии
- `SecurityConfig.java` - конфигурация безопасности
- `generate_certificates.ps1/sh` - генерация сертификатов
